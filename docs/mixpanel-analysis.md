## Executive Summary

4월 초 런칭 이후 약 10일간 **142 WAU / 주간 리텐션 48.76%**를 확보했고, 핵심으로 설계한 *판매처 비교*(`wishlist_add`)와 *특전 시각화*(`open_benefits`) 기능이 사용 빈도 1·2위를 기록해 제품 가설이 데이터로 검증됐다. 구매 퍼널의 병목은 **"페이지뷰 → 판매처 상세"(42.7%)**와 **"찜 → 구매 기록"(29.55%)** 두 지점으로 특정됐으며, 다음 이터레이션의 개선 포인트로 이어진다.

## 2. 리텐션 — 가장 강력한 지표

**7일 리텐션 48.76%**는 이 서비스에서 가장 주목할 숫자다.

- 일반 웹 서비스의 주간 리텐션은 10~20%대가 흔하다
- 48.76%는 사용자의 **절반**이 다음 주에 돌아온다는 뜻
- 서비스 성격과 완벽히 일치한다: 예약판매 기간 동안 *"새 판매처 열렸나? 마감 임박 건 있나?"*를 매일 확인하러 돌아오는 고빈도 체크인 도구이기 때문

### 해석

이 리텐션 수치는 다음 설계 가설을 데이터로 검증한다.

> "예약판매 사이클(D-14 ~ D-day)에 맞춘 고주기 체크인 도구로 설계했고, 사용자는 그 사이클에 맞춰 실제로 재방문한다."

포트폴리오 관점에서 이 수치가 핵심 스토리다.

## 3. 구매 퍼널 — 병목의 정답지

```
Page View ............... 178 (100%)
    ↓ 42.7%              ← 최대 이탈 (57.3% drop)
retailer_detail .......... 76
    ↓ 57.89%
wishlist_add ............. 44
    ↓ 29.55%
purchase_complete ........ 13

Overall conversion: 7.3%
```

### 단계별 해석

#### 1단계 → 2단계 (Page View → retailer_detail): 42.7%
가장 큰 이탈 구간. 해석은 두 가지다.

- **가설 A**: "그냥 구경하러 온 사람"이 많다. 팬덤 특성상 자연스럽다.
- **가설 B**: 판매처 카드 자체에서 이미 충분한 정보를 얻어 굳이 상세를 열지 않는다.

**개선 방향**: 카드 레벨에 *특전 썸네일*과 *D-day 배지*를 더 강하게 노출하면 두 방향 모두에 대응할 수 있다. 상세 진입 없이도 만족 가능하게 하거나(가설 B), 반대로 카드 수준에서 "더 보기" 욕구를 자극(가설 A)할 수 있다.

#### 2단계 → 3단계 (retailer_detail → wishlist_add): 57.89%
**매우 건강한 수치.** 상세를 연 사람의 절반 이상이 찜으로 전환한다. 상세 페이지가 구매 의사결정을 실제로 돕고 있다는 뜻이며, 이 구간은 지금 그대로 유지해도 무방하다.

#### 3단계 → 4단계 (wishlist_add → purchase_complete): 29.55%
**주의해서 읽어야 한다.** 이것은 *실제 구매 전환율*이 아니라 *앱 안에서 구매 완료를 기록한 비율*이다.

- 44명 찜 → 13명 기록 → 70%가 찜만 하고 기록은 안 함
- **가능성 1**: 실제로 구매를 안 함 (장바구니만 활용)
- **가능성 2**: 구매는 했지만 기록을 잊거나 귀찮음

현재 지표로는 이 둘을 구분할 수 없다. 5장에서 제안하는 `purchase_link_click` 이벤트를 추가하면 "실제 구매 의사"와 "기록 행위"를 분리 측정할 수 있다.

## 4. 기능 사용률 — 설계 의도 검증

| 순위 | 기능 | 30일 카운트 | 해석 |
|---|---|---|---|
| 1 | `wishlist_add` | **72** | 핵심 기능으로 설계한 것이 실제 1위 — 설계 의도 적중 |
| 2 | `open_benefits` | **51** | 미공포 갤러리가 2위 — 시각 자산 중심 설계의 정당성 입증 |
| 3 | `purchase_complete` | **23** | 기록 기능은 절반 이하. 위 퍼널의 29.55% 이탈과 일치 |
| 4 | `open_calendar` | **12** | 낮음. 마감 정보가 이미 카드/리스트에 충분히 노출돼 별도 캘린더 필요가 적을 수 있음 |
| 5 | `open_sync` | **4** | 가장 낮음. 발견성 문제 또는 단일 기기 사용이 지배적일 가능성 |

### 핵심 시사점

- **`wishlist_add`와 `open_benefits`의 1·2위는 이 서비스의 가장 중요한 데이터 포인트다.** 서비스 목적(3번 문서 *service-overview.md §2*)에서 정의한 "판매처 비교 + 특전 시각화"가 실제 사용자 행동으로 나타났음을 의미한다.
- **`open_sync`가 4회**인 것은 두 가지로 해석 가능하다.
  - 해석 A: 동기화 기능의 발견성이 부족 → UX 개선 여지
  - 해석 B: 대부분 사용자가 모바일 단일 기기만 사용 → 기능 우선순위 하향 가능
  - 이 구분은 `sync_complete` 같은 후속 이벤트 추가 없이는 불가능하다.
- **`open_calendar` 12회**는 기능 실패가 아니라 "메인 대시보드가 이미 마감 정보를 충분히 전달하고 있다"는 신호일 가능성이 높다. 메인에 D-day 정렬이 도입되어 있고, 리스트에서 마감 임박 건이 상단에 노출되기 때문이다.

## 5. 세션당 인터랙션 — 참여 깊이

평균 **6.4회/세션**, 피크는 4월 초 25~30회. 이는 "둘러보고 나가는" 앱이 아니라 **세션 안에서 여러 판매처를 비교·찜·기록하는 능동적 사용 패턴**을 보여준다.

48.76%의 리텐션과 묶어 읽으면, "자주 오고, 올 때마다 깊게 쓰는" 사용자 프로필이 그려진다. 이는 본 서비스가 가장 지향했던 사용 맥락(*service-overview.md §1 — "앨범 예약판매 기간에 매일 비교·결제·정산을 반복하는 고빈도 작업"*)과 정확히 일치한다.

## 7. 추가 이벤트 제안

현재 지표로 답할 수 없는 질문들을 해결하기 위해 다음 5개 이벤트의 추가를 제안한다. 각 이벤트는 *현재의 빈 구간*을 메우도록 설계됐다.

### 7.1 `purchase_link_click`
**발화 시점**: 판매처 상세 패널에서 "구매하러 가기" 외부 링크 클릭 시
**속성**:
- `retailer_id` (판매처 식별자)
- `version` (PHOTOBOOK / ID PASS / INVENTORY / POCAALBUM)
- `source` (list_card / detail_panel / benefits_modal)

**해결하는 질문**:
- 찜 → 구매 기록 이탈(29.55%)이 *"실제 미구매"* 때문인지 *"기록 누락"* 때문인지 분리 가능
- 어느 판매처가 실제로 가장 높은 구매 유도력을 가지는가
- 구매 퍼널에 "link_click"을 끼워 넣어 *실질 구매 전환*과 *앱 내 기록 전환*을 별도 지표로 관리

### 7.2 `version_tab_change`
**발화 시점**: 메인 대시보드에서 앨범 버전 탭 전환 시
**속성**:
- `from_version` / `to_version`
- `entry_method` (tab_click / filter_shortcut)

**해결하는 질문**:
- 4종 버전 중 어느 것이 사용자 관심이 가장 높은가
- 여러 버전을 비교하는 사용자 비율은 얼마인가 (1종만 보는 사람 vs. 멀티 비교 사용자)
- 드볼 완성을 위해 실제로 여러 버전을 오가는 행동이 일어나는가

### 7.3 `filter_apply` / `sort_change`
**발화 시점**: 판매처 리스트의 필터·정렬 옵션 변경 시
**속성**:
- `filter_type` (all / available / cart / purchased / closed)
- `sort_key` (benefits / deadline / price)
- `exclude_purchased` (boolean)

**해결하는 질문**:
- 사용자가 판매처를 고를 때 무엇을 가장 중요하게 여기는가 (특전·마감·가격)
- 필터·정렬 기능이 실제로 사용되는가, 아니면 기본 상태 그대로 소비되는가
- 기본 정렬 기준을 어느 것으로 둘지 데이터 기반으로 결정 가능

### 7.4 `benefit_modal_open`
**발화 시점**: 미공포 갤러리(`/benefits`)에서 특전 카드를 탭해 상세 모달을 열 때
**속성**:
- `benefit_id`
- `retailer_id`
- `collection_status` (owned / wishlisted / available / closed)

**해결하는 질문**:
- 어떤 미공포 포카가 가장 주목받는가 (인기 랭킹)
- 미공포 페이지에 들어온 사용자가 실제로 상세까지 파고드는가, 단순 그리드 스크롤인가
- 인기 특전이 실제로 `wishlist_add`로 이어지는가 (두 이벤트 간 funnel)

### 7.5 `debul_progress_view`
**발화 시점**: 사이드 패널의 드볼 진행도 영역이 뷰포트에 3초 이상 노출될 때 (intersection observer)
**속성**:
- `version`
- `progress_percentage`
- `bottleneck_item` (10종 포카 / 10종 씰 등)

**해결하는 질문**:
- 드볼 기능이 실제로 관찰되는가, 아니면 UI에만 존재하는가
- 드볼 진행도를 본 사용자가 그 직후 `wishlist_add` 행동으로 이어지는가 (기능이 구매 유도에 기여하는지)
- 어느 버전의 드볼을 시도하는 사용자가 가장 많은가

## 8. 다음 단계

1. **7장의 5개 이벤트 구현** — 우선순위는 `purchase_link_click` > `version_tab_change` > `filter_apply` 순 (구매 퍼널 해석의 정확도를 높이는 순서)
2. **구매 기록 리마인더 실험** — 찜 등록 후 N일 지나도 `purchase_complete` 이벤트가 없으면 가벼운 힌트(예: "구매하셨나요? 기록해 두면 드볼 진행도에 반영돼요") 노출 → A/B로 29.55% 구간 개선 여부 확인
3. **카드 레벨 정보 밀도 실험** — 판매처 카드에 특전 썸네일 1~2개를 추가 노출하는 변형을 시도해 1→2단계 이탈(42.7%)이 완화되는지 확인
4. **해외팬 획득 테스트** — 현재 국가 분포가 100% 한국인데, 서비스 자체는 해외 판매처 데이터를 풍부하게 가지고 있음. 영어 요약 뷰 또는 해외 팬덤 채널 공유 등으로 잠재 성장축 탐색
